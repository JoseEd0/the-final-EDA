\section*{Como estudiar este PDF}
Este documento esta escrito para repasar desde cero los problemas que si sirven
para el final: los concursos utiles del grupo Racso, mas los externos que
refuerzan exactamente los mismos temas. No hay codigo. En cada problema debes
aprender a escribir en papel: estructura, algoritmo, correctitud y complejidad.

Los temas que se repiten son:
\begin{itemize}[leftmargin=*]
\item RMQ, LCA y consultas en arboles estaticos.
\item Suffix array, LCP, busqueda y conteo de subcadenas.
\item Euler tour, DSU on tree, HLD, mascaras de paridad, centroid decomposition.
\item Dynamic connectivity offline con segment tree sobre tiempo y DSU rollback.
\item Dynamic trees con Link-Cut Trees o Euler-Tour Trees.
\end{itemize}

No se incluyen Proyecto 2/Efecto cascada ni Proyecto 3/BST implicito porque no
son prioridad para esta ruta.

\section{Static RMQ}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/694840/problem/A}

\subsection*{Resumen}
Se da un arreglo fijo $A[0..n-1]$ y muchas consultas $[l,r]$. Para cada consulta
se pide el minimo valor dentro del rango. El arreglo no cambia. Esa palabra,
\emph{static}, es lo que permite preprocesar fuerte.

\subsection*{Teoria desde cero}
RMQ significa \emph{Range Minimum Query}. Si el arreglo cambiara, se usaria un
segment tree. Pero aqui no cambia, entonces conviene guardar respuestas para
rangos especiales. La estructura mas importante para empezar es \textbf{sparse
table}. Guarda minimos de bloques de longitud potencia de dos:
\[
st[k][i] = \min(A[i], A[i+1], \ldots, A[i+2^k-1]).
\]
El bloque de longitud $2^k$ se parte en dos bloques de longitud $2^{k-1}$:
\[
st[k][i] = \min(st[k-1][i], st[k-1][i+2^{k-1}]).
\]

\subsection*{Algoritmo}
Preprocesa todos los $st[k][i]$. Para consultar $[l,r]$, toma
$len=r-l+1$ y $k=\lfloor\log_2 len\rfloor$. Usa dos bloques de longitud $2^k$:
uno empieza en $l$ y el otro termina en $r$:
\[
ans = \min(st[k][l], st[k][r-2^k+1]).
\]
Los bloques pueden solaparse.

\subsection*{Correctitud}
Los dos bloques cubren todo el intervalo $[l,r]$. El minimo global del intervalo
esta en al menos uno de los dos bloques. Si los bloques se solapan, no hay
problema porque el minimo es idempotente: $\min(x,x)=x$. Por eso tomar el minimo
entre los minimos de ambos bloques devuelve exactamente el minimo del rango.

\subsection*{Complejidad}
Preprocesamiento $\bigO(n\log n)$, memoria $\bigO(n\log n)$ y cada consulta
$\bigO(1)$. En teoria avanzada existe RMQ con $\bigO(n)$ espacio y $\bigO(1)$
query usando macro/micro bloques, pero sparse table es la respuesta limpia para
papel si no piden optimalidad estricta.

\section{Suffix array y LCP array}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/A}

\subsection*{Resumen}
Este problema es la base de todo el bloque de strings: construir o razonar con
el suffix array y el LCP array de una cadena.

\subsection*{Teoria desde cero}
Un sufijo de $S$ es $S[i..n-1]$. El suffix array $\SA$ es una permutacion de
posiciones tal que los sufijos quedan ordenados lexicograficamente:
\[
S[\SA[0]..] < S[\SA[1]..] < \cdots < S[\SA[n-1]..].
\]
El LCP array mide prefijos comunes entre sufijos consecutivos:
\[
\LCP[i]=\operatorname{lcp}(S[\SA[i-1]..],S[\SA[i]..]).
\]
Un caracter sentinela menor que todos, por ejemplo \texttt{\$}, evita problemas
cuando un sufijo es prefijo de otro.

\subsection*{Estructura y algoritmo}
El algoritmo de clase mas importante es Manber--Myers. Mantiene clases de
equivalencia para prefijos de longitud $2^k$. Inicialmente se ordena por primer
caracter. En la ronda $k$, el sufijo que empieza en $i$ se representa por:
\[
(rank[i], rank[i+2^k]).
\]
Ordenar esos pares equivale a ordenar por los primeros $2^{k+1}$ caracteres.
Luego se reasignan ranks. Al llegar a longitud al menos $n$, los ranks ordenan
sufijos completos.

\subsection*{Correctitud}
Si dos sufijos tienen iguales sus primeros $2^{k+1}$ caracteres, entonces sus
primeros $2^k$ caracteres coinciden y tambien coinciden los siguientes $2^k$.
Por eso el par de ranks describe correctamente el prefijo doble. Si los pares
difieren, el orden lexicografico del par decide el orden del prefijo completo.
Por induccion en $k$, cada ronda ordena por una longitud cada vez mayor, y al
superar $n$ queda el orden de sufijos.

\subsection*{Complejidad}
Hay $\lceil\log n\rceil$ rondas. Si cada ronda se ordena con counting/radix sort
sobre ranks, cuesta $\bigO(n)$ por ronda y $\bigO(n\log n)$ total. El LCP puede
calcularse con Kasai en $\bigO(n)$ usando el rank inverso de cada sufijo.

\section{Busqueda de subcadenas}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/B}

\subsection*{Resumen}
El objetivo tipico es responder si un patron $P$ aparece en un texto fijo $S$,
cuantas veces aparece, o listar sus posiciones.

\subsection*{Teoria desde cero}
Un patron aparece en $S$ si existe un sufijo de $S$ que empieza con $P$. En el
suffix array, los sufijos que empiezan con el mismo prefijo forman un intervalo
contiguo. Esta es la propiedad central.

\subsection*{Algoritmo}
Construye $\SA$ de $S$. Para una consulta con patron $P$:
\begin{enumerate}[leftmargin=*]
\item Busca con binary search el primer sufijo que es lexicograficamente mayor
o igual que $P$.
\item Busca el primer sufijo que ya es estrictamente mayor que cualquier cadena
que tenga prefijo $P$.
\item El intervalo entre ambos limites contiene todas las ocurrencias.
\end{enumerate}
Si el intervalo es vacio, el patron no aparece. Si tiene tamano $t$, aparece
$t$ veces. Las posiciones son los valores $\SA[i]$ dentro del intervalo.

\subsection*{Correctitud}
Todos los strings que comparten un prefijo $P$ aparecen juntos en orden
lexicografico: no puede haber entre ellos un sufijo que no tenga prefijo $P$,
porque eso romperia el orden en el primer caracter donde difiere. Las dos
busquedas binarias encuentran exactamente los bordes inferior y superior de ese
bloque. Por tanto, el intervalo es exactamente el conjunto de ocurrencias.

\subsection*{Complejidad}
Construccion $\bigO(n\log n)$ con Manber--Myers. Cada comparacion directa cuesta
$\bigO(|P|)$, asi que query $\bigO(|P|\log n)$ en la version simple. Con LCP y
RMQ se puede mejorar, pero para teoria de examen basta explicar el intervalo.

\section{Contando subcadenas}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/C}

\subsection*{Resumen}
El nombre puede apuntar a contar ocurrencias de patrones o contar subcadenas con
alguna condicion. La base para resolverlo es entender que suffix array transforma
problemas de subcadenas en intervalos de sufijos.

\subsection*{Teoria desde cero}
Toda subcadena de $S$ es prefijo de algun sufijo. Si quieres contar ocurrencias
de una subcadena/patron $P$, debes contar cuantos sufijos empiezan con $P$. En
$\SA$, eso es el tamano de un intervalo contiguo.

\subsection*{Algoritmo para contar ocurrencias}
Construye $\SA$. Para cada patron $P$, encuentra el intervalo $[L,R]$ de sufijos
que tienen prefijo $P$ con dos busquedas binarias. La cantidad de ocurrencias es:
\[
R-L+1.
\]
Si el intervalo no existe, la respuesta es $0$.

\subsection*{Si la pregunta es contar subcadenas distintas}
Entonces se usa la formula del problema siguiente:
\[
\frac{n(n+1)}2-\sum_i \LCP[i].
\]
La diferencia es: contar ocurrencias de un patron usa intervalos; contar
subcadenas distintas usa el total menos duplicados medidos por LCP.

\subsection*{Complejidad}
Preprocesar $\SA$ cuesta $\bigO(n\log n)$ o $\bigO(n)$ con DC3/SA-IS. Cada
consulta de conteo cuesta $\bigO(|P|\log n)$ en la version simple. La memoria es
$\bigO(n)$ para $\SA$ y ranks, mas lo necesario para LCP si se usa.

\section{Subcadenas diferentes}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/D}

\subsection*{Resumen}
Se pide contar cuantas subcadenas distintas tiene una cadena.

\subsection*{Teoria desde cero}
Un string de longitud $n$ tiene $\frac{n(n+1)}2$ subcadenas contando repetidas.
Cada subcadena es un prefijo de algun sufijo. Si procesas sufijos en orden
lexicografico, el sufijo actual aporta nuevos prefijos salvo los que ya compartia
con el sufijo anterior.

\subsection*{Formula}
El sufijo que empieza en $\SA[i]$ tiene longitud $n-\SA[i]$, asi que tiene
$n-\SA[i]$ prefijos. Pero sus primeros $\LCP[i]$ prefijos ya aparecieron como
prefijos del sufijo anterior en orden lexicografico. Entonces aporta:
\[
(n-\SA[i])-\LCP[i].
\]
Sumando sobre todos los sufijos:
\[
\#distinct = \frac{n(n+1)}2-\sum_i \LCP[i].
\]

\subsection*{Correctitud}
Cada subcadena se cuenta cuando aparece por primera vez como prefijo de algun
sufijo en orden lexicografico. Si un prefijo del sufijo actual ya aparecio antes,
entonces aparece en algun sufijo vecino con prefijo comun; el LCP con el sufijo
anterior captura exactamente la cantidad de prefijos que no son nuevos. Por eso
restar la suma de LCP elimina duplicados una sola vez.

\subsection*{Complejidad}
Construir $\SA$ y $\LCP$ domina el costo: $\bigO(n\log n)$ con Manber--Myers y
Kasai, o $\bigO(n)$ con construccion lineal. El conteo final es $\bigO(n)$.

\section{Subcadena en comun mas larga}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/E}

\subsection*{Resumen}
Dadas dos cadenas $A$ y $B$, se pide la longitud o contenido de la subcadena mas
larga que aparece en ambas.

\subsection*{Teoria desde cero}
Si una cadena $X$ aparece en $A$ y en $B$, entonces existen dos sufijos, uno de
$A$ y uno de $B$, que tienen a $X$ como prefijo comun. El problema se vuelve
buscar el mayor LCP entre sufijos de origen distinto.

\subsection*{Algoritmo}
Concatena:
\[
T = A \# B \$,
\]
donde $\#$ y $\$$ no aparecen en las cadenas. Construye $\SA$ y $\LCP$ de $T$.
Recorre posiciones consecutivas del suffix array. Si $\SA[i-1]$ viene de $A$ y
$\SA[i]$ viene de $B$, o viceversa, actualiza la respuesta con $\LCP[i]$.

\subsection*{Correctitud}
Todos los sufijos que comparten un prefijo $X$ aparecen consecutivos en $\SA$.
Si en ese bloque hay al menos un sufijo de $A$ y uno de $B$, entonces dentro del
bloque existe algun par adyacente de origen distinto. El LCP de ese par mide una
subcadena comun de longitud al menos $|X|$. Por tanto el maximo entre pares
adyacentes de origen distinto da la subcadena comun mas larga.

\subsection*{Complejidad}
Construccion de $\SA$ y $\LCP$: $\bigO(n\log n)$ o $\bigO(n)$ segun algoritmo.
Recorrido final: $\bigO(n)$. Memoria $\bigO(n)$.

\section{Ordenando subcadenas}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/F}

\subsection*{Resumen}
Se deben comparar u ordenar subcadenas de un mismo texto. El problema combina
suffix array, LCP y RMQ.

\subsection*{Teoria desde cero}
Para comparar $S[l_1..r_1]$ y $S[l_2..r_2]$, necesitas saber el primer punto
donde difieren. Eso es el LCP entre los sufijos $S[l_1..]$ y $S[l_2..]$,
recortado por las longitudes de las subcadenas.

\subsection*{Estructura}
Construye:
\begin{itemize}[leftmargin=*]
\item suffix array $\SA$;
\item rank inverso: posicion de cada sufijo en $\SA$;
\item LCP array;
\item RMQ sobre LCP para obtener LCP entre dos sufijos arbitrarios.
\end{itemize}
Si los sufijos de $l_1$ y $l_2$ estan en posiciones $a<b$ en $\SA$:
\[
\operatorname{lcp}(l_1,l_2)=\min(\LCP[a+1],\ldots,\LCP[b]).
\]

\subsection*{Comparador teorico}
Sea $x$ el LCP de los sufijos y $len_1,len_2$ las longitudes de las subcadenas.
Si $x\ge \min(len_1,len_2)$, entonces una subcadena es prefijo de la otra y
decide la longitud. Si no, se compara $S[l_1+x]$ con $S[l_2+x]$.

\subsection*{Complejidad}
Preprocesamiento $\bigO(n\log n)$ con sparse table sobre LCP. Cada comparacion
$\bigO(1)$. Ordenar $q$ subcadenas cuesta $\bigO(q\log q)$ comparaciones.

\section{Bordes}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/G}

\subsection*{Resumen}
Un borde de una cadena es una cadena que es prefijo y sufijo a la vez. Por
ejemplo, en \texttt{ababa}, \texttt{a} y \texttt{aba} son bordes.

\subsection*{Teoria desde cero}
La herramienta mas directa es la funcion prefijo de KMP. Para cada posicion $i$:
\[
\pi[i] = \text{longitud del mayor borde de } S[0..i].
\]
Si conoces el mayor borde de todo el string, los bordes menores se obtienen
siguiendo bordes del borde:
\[
k=\pi[n-1],\quad k=\pi[k-1],\quad \ldots
\]

\subsection*{Algoritmo}
Calcula $\pi$. Empieza con $k=\pi[n-1]$. Mientras $k>0$, registra $k$ como una
longitud de borde y reemplaza $k$ por $\pi[k-1]$. Al final, si el problema pide
tambien el string completo como borde impropio/propio, se maneja segun definicion.

\subsection*{Correctitud}
Si $X$ es borde de $S$, entonces $X$ es prefijo de $S$ y tambien sufijo. El mayor
borde esta en $\pi[n-1]$. Cualquier borde menor de $S$ debe ser borde de ese
mayor borde; por eso seguir la cadena de $\pi$ enumera todos los bordes.

\subsection*{Complejidad}
KMP prefix function cuesta $\bigO(n)$. Enumerar bordes cuesta proporcional al
numero de bordes, a lo mucho $\bigO(n)$.

\section{Refran}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/H}

\subsection*{Resumen}
Este tipo de problema suele pedir una subcadena repetida que maximiza alguna
funcion, por ejemplo longitud por frecuencia. La estructura natural es suffix
array + LCP.

\subsection*{Teoria desde cero}
Una subcadena aparece $k$ veces si existen $k$ sufijos que tienen esa subcadena
como prefijo. En el suffix array, esos $k$ sufijos aparecen en un bloque
consecutivo. La longitud comun minima del bloque se mide por el minimo de los
LCP internos.

\subsection*{Estructura}
Interpreta el arreglo LCP como un histograma. Una altura $h$ mantenida sobre un
intervalo de LCP significa que hay varios sufijos consecutivos que comparten un
prefijo de longitud al menos $h$. Si el intervalo tiene $w$ barras de LCP, cubre
$w+1$ sufijos, entonces la frecuencia es $w+1$.

\subsection*{Algoritmo conceptual}
Usa pila monotona sobre LCP para cada posicion como altura minima de un intervalo
maximal. Para cada altura $h$, calcula cuantos sufijos consecutivos tienen LCP
al menos $h$, y evalua la funcion pedida, por ejemplo:
\[
h \cdot (cantidad\ de\ sufijos).
\]

\subsection*{Complejidad}
Construir $\SA+\LCP$ cuesta $\bigO(n\log n)$ o $\bigO(n)$. La pila monotona
recorre LCP en $\bigO(n)$. Memoria $\bigO(n)$.

\section{Substring periodica}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/695758/problem/I}

\subsection*{Resumen}
Una cadena es periodica si se puede describir por repeticiones de un patron mas
corto. Para substrings, se necesita verificar periodicidad de muchas porciones
de un texto.

\subsection*{Teoria desde cero}
Una cadena $X$ de longitud $m$ tiene periodo $p$ si:
\[
X[i]=X[i+p]\quad \text{para todo }0\le i<m-p.
\]
Si $p$ divide a $m$ y el periodo cubre todo, $X$ es repeticion de un bloque de
longitud $p$. Herramientas utiles: prefix function, Z-function y LCP entre
sufijos.

\subsection*{Estructura}
Para muchas comparaciones de substrings, usa suffix array + LCP + RMQ. Eso
permite responder en $\bigO(1)$ si dos substrings largas son iguales: basta con
ver si el LCP de sus posiciones alcanza la longitud requerida.

\subsection*{Algoritmo conceptual}
Para probar si una substring tiene periodo $p$, verifica que su prefijo de
longitud $m-p$ coincide con la substring desplazada $p$ posiciones. Eso se
responde con LCP:
\[
\operatorname{lcp}(l,l+p)\ge m-p.
\]
Para buscar periodos, se prueban candidatos segun divisores o segun estructura
del problema.

\subsection*{Complejidad}
Con preprocesamiento $\SA+\LCP+RMQ$, cada prueba de igualdad de substrings cuesta
$\bigO(1)$. La complejidad total depende de cuantos periodos candidatos se
prueben. Este es de los mas avanzados; asegura primero SA, LCP y RMQ.

\section{Tree and Queries}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/697527/problem/A}

\subsection*{Resumen}
Cada nodo tiene un color. Cada consulta pregunta, en el subarbol de un nodo
$v$, cuantos colores aparecen al menos $k$ veces.

\subsection*{Teoria desde cero}
Euler tour convierte el subarbol de $v$ en un intervalo, pero la pregunta no es
un simple minimo/suma: pide frecuencias de colores dentro del subarbol. La
tecnica natural es \textbf{DSU on tree} o \textbf{small-to-large}.

\subsection*{Estructura}
Para cada color $c$, guarda $cntColor[c]$. Para responder ``cuantos colores
tienen frecuencia al menos $k$'', guarda tambien informacion por frecuencia,
por ejemplo un Fenwick sobre $cntFreq[f]$, donde $cntFreq[f]$ es cuantos colores
aparecen exactamente $f$ veces.

\subsection*{Algoritmo}
Elige en cada nodo un hijo pesado: el de mayor subarbol. Procesa hijos ligeros y
borra su informacion. Procesa el hijo pesado y conserva su informacion. Luego
agrega manualmente todos los nodos de los hijos ligeros y el nodo actual. En ese
momento la bolsa contiene exactamente el subarbol actual, asi que respondes sus
consultas.

\subsection*{Correctitud}
Los subarboles de los hijos de $v$ son disjuntos y junto con $v$ forman todo el
subarbol de $v$. Al conservar el hijo pesado y agregar los ligeros, la bolsa
queda exactamente igual al subarbol completo. Como las consultas se responden en
ese instante, las frecuencias son correctas.

\subsection*{Complejidad}
Cada nodo se re-agrega solo cuando pertenece a un subarbol ligero. En una ruta
raiz-hoja hay $\bigO(\log n)$ aristas ligeras, porque al subir por una ligera el
tamano se duplica. Cota segura: $\bigO(n\log n)$ mas el costo de updates de
frecuencias.

\section{Arpa's letter-marked tree}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/697527/problem/B}

\subsection*{Resumen}
Problema de arbol con letras donde se razona si caminos o conjuntos de letras
pueden formar palindromos.

\subsection*{Teoria desde cero}
Una multiconjunto de letras puede reordenarse como palindromo si a lo mucho una
letra aparece cantidad impar de veces. Esta condicion se representa con una
mascara de bits: bit $j=1$ si la letra $j$ aparece impar cantidad de veces.
Una mascara es valida para palindromo si tiene cero o un bit prendido.

\subsection*{Estructura}
Define $maskRoot[v]$ como xor de letras desde la raiz hasta $v$. La mascara del
camino $u\leadsto v$ se obtiene combinando mascaras de prefijo y ajustando el
LCA. Conceptualmente:
\[
mask(u,v)=maskRoot[u]\oplus maskRoot[v]\oplus letter[\lca(u,v)].
\]
Segun si las letras estan en vertices o aristas, el ajuste cambia, pero la idea
es xor de prefijos.

\subsection*{Algoritmo}
Para contar pares o caminos validos, se mantiene un contador de mascaras. Una
mascara $x$ puede emparejarse con:
\[
x \quad \text{o} \quad x\oplus 2^b
\]
para cualquier bit $b$. Esto cubre cero o una paridad impar. En subarboles, se
combina con DSU on tree para mantener conteos de mascaras eficientemente.

\subsection*{Correctitud}
El xor cancela letras que aparecen dos veces. Por tanto la mascara resultante
codifica exactamente que letras aparecen impar cantidad de veces en el camino.
La condicion de palindromo es equivalente a que esa mascara tenga a lo mucho un
bit encendido.

\subsection*{Complejidad}
Con alfabeto pequeno, revisar todas las mascaras que difieren en un bit cuesta
$\bigO(\Sigma)$ por insercion/consulta. Con DSU on tree, cota segura
$\bigO(n\log n\cdot \Sigma)$.

\section{Laminar Family}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/697527/problem/C}

\subsection*{Resumen}
Una familia laminar es una coleccion de conjuntos donde dos conjuntos
cualesquiera son disjuntos o uno contiene al otro. En arboles, los subarboles
tienen naturalmente esta forma.

\subsection*{Teoria desde cero}
Haz un DFS y asigna tiempos $\tin[v]$ y $\tout[v]$. El subarbol de $v$ se vuelve
el intervalo:
\[
[\tin[v],\tout[v]].
\]
Un nodo $u$ es ancestro de $v$ si:
\[
\tin[u]\le \tin[v]\le \tout[u].
\]
Dos subarboles en un arbol no se cruzan parcialmente: o son disjuntos, o uno
contiene al otro.

\subsection*{Algoritmo conceptual}
Representa cada conjunto/subarbol por su intervalo Euler. Para verificar si una
familia es laminar, ordena intervalos por inicio y usa una pila de intervalos
activos. Cuando entra un intervalo nuevo, debe estar contenido en el ultimo
activo o empezar despues de que los activos incompatibles terminen. Si hay
interseccion parcial, no es laminar.

\subsection*{Correctitud}
La transformacion DFS preserva inclusion de subarboles como inclusion de
intervalos. Las unicas relaciones validas en una familia laminar son inclusion o
disjuncion. Por eso detectar cruces parciales entre intervalos es equivalente a
detectar violaciones de laminaridad.

\subsection*{Complejidad}
DFS $\bigO(n)$. Ordenamiento $\bigO(m\log m)$ si hay $m$ conjuntos. La verificacion
con pila es $\bigO(m)$.

\section{Xenia and Tree}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/697527/problem/D}

\subsection*{Resumen}
Problema clasico: hay vertices rojos en un arbol. Se actualiza un vertice a rojo
y se pregunta la distancia al rojo mas cercano.

\subsection*{Teoria desde cero}
Si recalculas BFS desde cada consulta, es demasiado lento. La estructura natural
es \textbf{centroid decomposition}. Un centroide es un nodo cuya eliminacion
parte el arbol en componentes de tamano a lo mucho la mitad. Al descomponer
recursivamente, cada nodo queda asociado a $\bigO(\log n)$ centroides ancestros.

\subsection*{Estructura}
Para cada centroide $c$, guarda:
\[
best[c]=\min \dist(c,r)
\]
entre todos los rojos $r$ ya activados. Para cada nodo $v$, guarda la lista de
centroides ancestros y la distancia de $v$ a cada uno.

\subsection*{Algoritmo}
Al pintar $v$ rojo, para cada centroide ancestro $c$ de $v$, actualiza:
\[
best[c]=\min(best[c],\dist(v,c)).
\]
Para consultar $u$, calcula:
\[
\min_c best[c]+\dist(u,c),
\]
donde $c$ recorre los centroides ancestros de $u$.

\subsection*{Correctitud}
Para cualquier rojo $r$ y consulta $u$, existe un nivel de la descomposicion de
centroides donde los caminos desde $u$ y $r$ se encuentran representados por un
centroide comun $c$. La distancia via ese centroide se considera como
$best[c]+\dist(u,c)$. Al tomar minimo sobre todos los centroides ancestros, se
captura el rojo mas cercano.

\subsection*{Complejidad}
Cada nodo tiene $\bigO(\log n)$ centroides ancestros. Update y query cuestan
$\bigO(\log n)$ si las distancias a centroides estan precomputadas. Construccion
$\bigO(n\log n)$.

\section{Beauty and The Tree}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/697527/problem/E}

\subsection*{Resumen}
Problema de arbol mas avanzado, asociado a DP/probabilidad o conteos sobre
subarboles. No es tan central como Tree and Queries o Xenia, pero entrena
razonamiento de arboles.

\subsection*{Teoria desde cero}
Para cualquier DP en arbol, primero se enraiza. Luego se define que informacion
calcula cada nodo a partir de sus hijos. Si la respuesta de un nodo depende solo
de su subarbol, un DFS postorden basta. Si depende de todo el arbol visto desde
ese nodo, se necesita rerooting.

\subsection*{Estructura}
Guarda:
\begin{itemize}[leftmargin=*]
\item padre y profundidad;
\item tamano de subarbol;
\item valores DP que suben desde hijos;
\item si hace falta rerooting, valores prefijo/sufijo de contribuciones de hijos.
\end{itemize}

\subsection*{Algoritmo conceptual}
Primero calcula informacion local de subarboles. Luego, si la query pide
respuesta global por nodo, realiza una segunda pasada para pasar contribuciones
del padre hacia hijos. Este patron se llama rerooting DP.

\subsection*{Correctitud}
El postorden es correcto porque al procesar un nodo ya se conocen todos sus
hijos. El rerooting es correcto porque separa la contribucion del hijo actual de
la contribucion del resto del arbol, permitiendo reconstruir la respuesta como
si cada hijo fuera raiz.

\subsection*{Complejidad}
Normalmente $\bigO(n)$ o $\bigO(n\log n)$ segun la combinacion de estados. Para
examen, lo importante es justificar que cada arista se procesa cantidad constante
de veces.

\section{Connect and Disconnect}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/698987/problem/A}

\subsection*{Resumen}
Se agregan y eliminan aristas, y se hacen consultas de conectividad. Como hay
eliminaciones, DSU normal no basta.

\subsection*{Teoria desde cero}
DSU responde conectividad incremental: solo agregar aristas. Si borras aristas,
DSU no sabe separar componentes. La solucion offline convierte cada arista en
intervalos de vida. Luego procesa el tiempo con un segment tree.

\subsection*{Estructura}
Usa:
\begin{itemize}[leftmargin=*]
\item mapa de aristas activas para saber inicio de intervalo;
\item segment tree sobre tiempos $1..q$;
\item DSU con rollback, sin path compression;
\item pila de cambios para deshacer unions.
\end{itemize}

\subsection*{Algoritmo}
Cada arista activa en $[l,r)$ se inserta en $\bigO(\log q)$ nodos del segment
tree temporal. Recorre el segment tree en DFS. Al entrar a un nodo, une todas
sus aristas. Si es hoja, responde las consultas de ese tiempo. Al salir, revierte
el DSU al estado anterior.

\subsection*{Correctitud}
En una hoja $t$, el camino desde la raiz del segment tree hasta esa hoja contiene
exactamente todos los nodos temporales cuyos intervalos cubren $t$. Por
construccion, esos nodos contienen todas y solo las aristas activas en $t$. Por
eso el DSU representa el grafo correcto para responder conectividad.

\subsection*{Complejidad}
Cada intervalo va a $\bigO(\log q)$ nodos. Cada union/find cuesta $\bigO(\log n)$
sin compresion o casi constante en practica con union by size. Cota segura:
\[
\bigO((q+m)\log q \log n).
\]

\section{Addition on Segments}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/698987/problem/B}

\subsection*{Resumen}
Problema de operaciones activas en segmentos. Aunque no sea grafo, entrena el
mismo patron mental que offline dynamic connectivity: objetos que viven en
intervalos.

\subsection*{Teoria desde cero}
Si una operacion afecta un rango de tiempos o posiciones, puedes descomponer ese
rango en nodos de un segment tree. Asi evitas aplicar la operacion uno por uno
en cada punto.

\subsection*{Estructura}
Segment tree sobre el eje relevante: tiempo, posiciones o indices. Cada nodo
guarda las operaciones que cubren completamente su intervalo. Si el problema
requiere deshacer, la estructura interna debe soportar rollback.

\subsection*{Algoritmo}
Para cada operacion de adicion sobre $[l,r]$, insertala en el segment tree en
$\bigO(\log n)$ nodos. Recorre el arbol manteniendo el estado acumulado. En hojas
respondes el valor final o la consulta correspondiente. Al volver, deshaces lo
que agregaste si el estado no es persistente.

\subsection*{Correctitud}
Todo punto $x$ pertenece exactamente a los nodos del camino raiz-hoja que lo
cubren. Una operacion insertada en nodos que particionan $[l,r]$ estara activa
exactamente para los puntos de ese rango. Por eso al llegar a una hoja, el estado
incluye todas y solo las operaciones relevantes.

\subsection*{Complejidad}
Cada intervalo se guarda en $\bigO(\log n)$ nodos. Si cada aplicar/deshacer cuesta
$T$, el total es $\bigO(Q\log n\cdot T)$.

\section{Visitas}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/699336/problem/A}

\subsection*{Resumen}
Es el problema tipo \emph{Milk Visits}. Hay un arbol de granjas; cada nodo tiene
un tipo de vaca. Cada consulta da dos nodos $a,b$ y un tipo $c$. Se pregunta si
en el camino entre $a$ y $b$ existe al menos un nodo de tipo $c$.

\subsection*{Teoria desde cero}
Una solucion general con HLD podria consultar colores en caminos, pero seria mas
pesada. La observacion clasica de Milk Visits es por componentes de igual color.
Si dos nodos $a$ y $b$ estan en la misma componente maximal de nodos del mismo
tipo, entonces todo el camino entre ellos tiene ese mismo tipo. Si el tipo
preferido $c$ es distinto, no habra ningun $c$ en ese camino.

\subsection*{Estructura}
Comprime componentes conectadas de nodos con el mismo tipo. Asigna a cada nodo
un id de componente. Para una consulta $(a,b,c)$:
\begin{itemize}[leftmargin=*]
\item si $comp[a]\ne comp[b]$, el camino cruza al menos dos componentes de tipos
distintos, y se puede satisfacer cualquier consulta cuyo criterio sea la version
clasica de Milk Visits;
\item si $comp[a]=comp[b]$, todo el camino tiene el tipo de esa componente, y la
respuesta es si ese tipo coincide con $c$.
\end{itemize}

\subsection*{Correctitud}
Una componente maximal de igual tipo contiene exactamente nodos conectados por
caminos de ese tipo. Si $a$ y $b$ estan en la misma componente, el camino unico
entre ellos no sale de la componente, por tanto todos sus nodos tienen el mismo
tipo. Si estan en componentes distintas, el camino cambia de tipo al menos una
vez; en la version clasica, eso garantiza satisfaccion salvo cuando el camino
entero es de un unico tipo distinto al pedido.

\subsection*{Complejidad}
DFS para construir componentes: $\bigO(n)$. Cada consulta: $\bigO(1)$. Memoria:
$\bigO(n)$.

\section{Distancia unitaria}
\textbf{Link.} \url{https://codeforces.com/group/AmklRC8lhZ/contest/694842/problem/A}

\subsection*{Resumen}
Se da un arbol enraizado en $1$. Cada consulta da un conjunto de vertices. Se
pregunta si existe un camino desde la raiz hasta algun nodo $u$ tal que cada
vertice de la consulta este en ese camino o a distancia $1$ de algun vertice del
camino.

\subsection*{Teoria desde cero}
Si un vertice $v$ esta a distancia $1$ de un camino raiz-$u$, entonces su padre
puede estar en el camino. Por eso se transforma cada vertice consultado en su
padre (excepto la raiz, que se queda igual). Luego la pregunta se vuelve: estan
todos esos vertices transformados sobre un mismo camino desde la raiz?

\subsection*{Estructura}
Preprocesa:
\begin{itemize}[leftmargin=*]
\item profundidad;
\item padre inmediato;
\item tiempos Euler $\tin,\tout$ para ancestro;
\item opcionalmente LCA.
\end{itemize}
Un nodo $x$ es ancestro de $y$ si:
\[
\tin[x]\le \tin[y]\le \tout[x].
\]

\subsection*{Algoritmo}
Para cada consulta, reemplaza cada vertice $v$ por $parent[v]$ si $v\ne 1$.
Elige el vertice transformado mas profundo, llamalo $deep$. La respuesta es
YES si todos los vertices transformados son ancestros de $deep$. Si alguno no
lo es, no existe un unico camino raiz-$u$ que los cubra.

\subsection*{Correctitud}
Despues de subir cada vertice a su padre, la condicion de estar en el camino o a
distancia $1$ se convierte en estar sobre el camino. Si todos los transformados
estan en un camino raiz-$u$, el mas profundo de ellos puede actuar como extremo
o estar antes del extremo, y todos los demas deben ser ancestros de el. Si alguno
no es ancestro del mas profundo, se ramifica hacia otro lado y ningun camino
simple desde la raiz puede contenerlos a todos.

\subsection*{Complejidad}
DFS inicial $\bigO(n)$. Cada consulta cuesta proporcional a su tamano. Como la
suma de tamanos de consultas esta acotada, el total es $\bigO(n+\sum k_i)$.

\section{Imperial roads}
\textbf{Link.} \url{https://codeforces.com/gym/101889/problem/I}

\subsection*{Resumen}
Se tiene un grafo ponderado. Para cada consulta, una arista debe estar incluida
en el arbol generador. Se pide el minimo peso posible.

\subsection*{Teoria desde cero}
Primero construye un MST con Kruskal. Si fuerzas una arista que no esta en el
MST, al agregarla aparece un ciclo. Para volver a tener un arbol, debes eliminar
una arista de ese ciclo. Para minimizar el costo, eliminas la arista mas pesada
del camino dentro del MST.

\subsection*{Estructura}
Necesitas responder maximo peso de arista en el camino entre dos nodos del MST.
Usa LCA con binary lifting augmentado:
\begin{itemize}[leftmargin=*]
\item $up[v][j]$: ancestro $2^j$ de $v$;
\item $mx[v][j]$: maximo peso en el salto de $v$ a $up[v][j]$.
\end{itemize}

\subsection*{Algoritmo}
Sea $W$ el peso del MST. Para consulta $(u,v,w)$:
\[
ans = W + w - \operatorname{maxEdge}(u,v).
\]
Para calcular $\operatorname{maxEdge}$, iguala profundidades subiendo el nodo
mas profundo y acumulando maximos; luego sube ambos nodos juntos hasta llegar al
LCA.

\subsection*{Correctitud}
Agregar $(u,v,w)$ al MST crea un ciclo unico. Todo arbol que incluya esa arista
y use las demas del MST salvo una debe eliminar una arista del camino $u-v$.
Eliminar la mas pesada minimiza el nuevo peso. La propiedad del ciclo de MST
respalda que la arista mas pesada del ciclo es la candidata natural a remover.

\subsection*{Complejidad}
Kruskal $\bigO(m\log m)$. Preprocesamiento LCA $\bigO(n\log n)$. Cada consulta
$\bigO(\log n)$. Memoria $\bigO(n\log n)$.

\section{Fools and Roads}
\textbf{Link.} \url{https://codeforces.com/problemset/problem/191/C}

\subsection*{Resumen}
Se da un arbol y muchas rutas entre pares de vertices. Para cada arista, se pide
cuantas rutas pasan por ella.

\subsection*{Teoria desde cero}
Actualizar cada camino arista por arista seria lento. Como todas las rutas se
conocen antes de responder, se usa diferencia en arbol. Es la version arborea de
un difference array.

\subsection*{Algoritmo}
Para cada ruta $(u,v)$:
\[
delta[u]++,\quad delta[v]++,\quad delta[\lca(u,v)]-=2.
\]
Luego haces DFS postorden. Para cada hijo $x$, sumas sus acumulados hacia el
padre. El valor final acumulado en $x$ es la cantidad de rutas que pasan por la
arista $(parent[x],x)$.

\subsection*{Correctitud}
Una ruta $u-v$ sube desde $u$ hasta $a=\lca(u,v)$ y baja hasta $v$. Los $+1$ en
$u$ y $v$ hacen que al acumular suban dos contribuciones. Restar $2$ en el LCA
cancela ambas antes de que pasen por encima del LCA. Asi se marcan exactamente
las aristas de la ruta y ninguna otra. Por linealidad, funciona para todas las
rutas juntas.

\subsection*{Complejidad}
Preprocesar LCA $\bigO(n\log n)$. Cada ruta $\bigO(\log n)$. DFS final
$\bigO(n)$. Total $\bigO((n+q)\log n)$.

\section{Frequency of String}
\textbf{Link.} \url{https://codeforces.com/problemset/problem/963/D}

\subsection*{Resumen}
Para un texto fijo y consultas $(k,P)$, se pide la minima longitud de una
subcadena que contenga al patron $P$ al menos $k$ veces.

\subsection*{Teoria desde cero}
Si conoces todas las posiciones donde aparece $P$:
\[
p_1<p_2<\cdots<p_t,
\]
entonces una ventana con $k$ ocurrencias consecutivas desde $p_i$ hasta
$p_{i+k-1}$ tiene longitud:
\[
p_{i+k-1}-p_i+|P|.
\]
La respuesta es el minimo sobre $i$.

\subsection*{Estructura}
Para muchos patrones, usa Aho-Corasick: insertas todos los patrones en un trie,
construyes enlaces de fallo y recorres el texto una vez para recolectar
ocurrencias. Alternativamente, suffix array permite encontrar el intervalo de
sufijos que empiezan con $P$, pero luego hay que ordenar posiciones por aparicion
en el texto.

\subsection*{Correctitud}
Cualquier subcadena que contiene $k$ ocurrencias tiene una primera ocurrencia y
una k-esima ocurrencia. Para contener la k-esima debe terminar al menos en
$p_{i+k-1}+|P|-1$. Tomar exactamente desde $p_i$ hasta ese final alcanza esa
longitud. Por tanto revisar ocurrencias consecutivas es suficiente y optimo.

\subsection*{Complejidad}
Aho-Corasick se construye en $\bigO(L)$ o $\bigO(L|\Sigma|)$ segun transiciones,
donde $L$ es suma de longitudes. El recorrido del texto es $\bigO(n)$ mas
ocurrencias reportadas. Para cada patron, calcular ventanas cuesta lineal en su
numero de ocurrencias.

\section{Yosupo - Dynamic Tree Subtree Add Subtree Sum}
\textbf{Link.} \url{https://judge.yosupo.jp/problem/dynamic_tree_subtree_add_subtree_sum}

\subsection*{Resumen}
Arbol dinamico con link/cut, update de suma en subarbol y query de suma de
subarbol.

\subsection*{Teoria desde cero}
HLD clasico no basta porque el arbol cambia. Una estructura natural es
Euler-Tour Tree: representar cada arbol como una secuencia Euler guardada en un
BST balanceado implicito. Cortar y enlazar se vuelven operaciones de split y
merge sobre secuencias.

\subsection*{Estructura}
Cada nodo del BST guarda:
\begin{itemize}[leftmargin=*]
\item tamano del segmento;
\item suma del segmento;
\item lazy add pendiente;
\item punteros para split/merge.
\end{itemize}
El subarbol de un nodo se representa como un intervalo bajo una orientacion de
raiz adecuada.

\subsection*{Algoritmo}
Para update de subarbol, aisla el intervalo del subarbol con splits, aplica lazy
add y vuelve a unir. Para query, aisla el intervalo, lee la suma agregada y
vuelve a unir. Link y cut modifican la secuencia Euler con split/merge.

\subsection*{Correctitud}
El invariante es que la secuencia Euler representa exactamente el arbol actual.
Los splits no cambian el orden interno, solo aislan intervalos. Como el subarbol
corresponde a un intervalo, lazy add afecta exactamente sus vertices, y la suma
agregada corresponde exactamente al subarbol.

\subsection*{Complejidad}
Cada split/merge cuesta $\bigO(\log n)$ esperado/amortizado. Cada operacion usa
cantidad constante de splits/merges: $\bigO(\log n)$.

\section{SPOJ - Dynamic Tree Connectivity}
\textbf{Link.} \url{https://www.spoj.com/problems/DYNACON1/}

\subsection*{Resumen}
Bosque dinamico con operaciones link, cut y connected. Se pregunta si dos
vertices estan en la misma componente.

\subsection*{Teoria desde cero}
Si solo hubiera link, DSU bastaria. Con cut, DSU no puede separar. Online se
usan Link-Cut Trees o Euler-Tour Trees. Offline tambien se puede usar segment
tree sobre tiempo + DSU rollback.

\subsection*{Opcion online}
Link-Cut Tree mantiene preferred paths con splay trees. Permite link, cut y
find-root en $\bigO(\log n)$ amortizado. Dos nodos estan conectados si tienen la
misma raiz representada.

\subsection*{Opcion offline}
Cada arista tiene intervalos de vida entre link y cut. Se inserta en segment
tree temporal y se recorre con DSU rollback. En hojas, connected se responde
comparando representantes.

\subsection*{Correctitud}
En Link-Cut/Euler-Tour Trees, el invariante es que la estructura representa el
bosque actual. En rollback, el invariante es temporal: al llegar a la hoja $t$,
el DSU contiene exactamente las aristas activas en $t$.

\subsection*{Complejidad}
Online: $\bigO(\log n)$ amortizado por operacion. Offline: cada intervalo en
$\bigO(\log q)$ nodos, total aproximadamente $\bigO(q\log q\log n)$ con DSU
rollback sin compresion.
