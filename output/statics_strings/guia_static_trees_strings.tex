\section*{Como usar esta guia}
Esta guia esta hecha para estudiar rapido, no para leer pasivamente los slides.
El examen es a papel y los problemas son estilo Codeforces/Yosupo, pero la
respuesta esperada suele ser teorica: idea, estructura, pseudocodigo,
correctitud y complejidad. Por eso, para cada tema se enfatiza:
\begin{itemize}[leftmargin=*]
\item que problema reconoce la estructura;
\item que estructura usar;
\item que invariante/proof escribir;
\item que complejidad cerrar.
\end{itemize}

\section{Mapa mental para clasificar problemas}

\subsection{Primera pregunta: es array, arbol o string?}
Antes de pensar en algoritmos, clasifica el objeto principal:
\begin{center}
\begin{tabular}{p{0.24\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
Objeto & Frases tipicas & Estructuras candidatas \\
\midrule
Array estatico & rangos, minimo, maximo, no hay updates & Sparse table, RMQ lineal macro/micro \\
Arbol estatico & LCA, ancestro, camino, subarbol & Euler tour, binary lifting, RMQ, HLD conceptual \\
String texto fijo & buscar patrones, contar ocurrencias, subcadenas & Suffix array, LCP, RMQ sobre LCP \\
Muchos patrones & varios patrones contra un texto & Aho-Corasick o suffix array por intervalos \\
Substrings ordenadas & comparar/ordenar muchas subcadenas & SA + rank + LCP + RMQ \\
\bottomrule
\end{tabular}
\end{center}

\subsection{Segunda pregunta: hay updates?}
Los temas de esta carpeta son principalmente estaticos. Si el input no cambia,
preprocesa fuerte. Si hay updates, cuidado:
\begin{itemize}[leftmargin=*]
\item Array con updates: ya no basta sparse table; piensa segment tree.
\item Arbol fijo con path queries: HLD + segment tree puede aparecer.
\item String fijo con muchas queries: suffix array sirve.
\item String dinamico: no es el foco de estos slides.
\end{itemize}

\subsection{Tercera pregunta: la consulta es por rango, camino o prefijo?}
\begin{itemize}[leftmargin=*]
\item Rango en array: sparse table/RMQ.
\item Camino en arbol: LCA, HLD o transformar a RMQ.
\item Subarbol: Euler tour convierte a intervalo.
\item Patron en string: intervalo contiguo en suffix array.
\item Comparar dos substrings: LCP entre sufijos + RMQ sobre LCP.
\end{itemize}

\subsection{Checklist de respuesta de examen}
Para casi cualquier problema de estos temas escribe:
\begin{enumerate}[leftmargin=*]
\item \textbf{Modelo del problema.} Que se preprocesa y que se consulta.
\item \textbf{Estructura.} Sparse table, Cartesian tree, Euler tour, SA, LCP, etc.
\item \textbf{Invariante.} Que significa cada arreglo/tabla.
\item \textbf{Consulta.} Como se responde paso a paso.
\item \textbf{Correctitud.} Por que cubre todos los casos.
\item \textbf{Complejidad.} Preprocesamiento, query, espacio.
\end{enumerate}

\section{Diccionario fundamental desde cero}

Esta seccion existe porque en examen no basta decir ``uso sparse table'' o
``uso binary lifting''. Debes poder explicar que es la estructura, que guarda,
por que responde la consulta y por que su complejidad es la que dices.

\subsection{Preprocesamiento}
Preprocesar significa gastar tiempo antes de recibir o responder consultas para
construir informacion auxiliar. La idea central de las estructuras estaticas es:
si el arreglo, arbol o string no cambia, conviene calcular tablas una sola vez
para responder muchas preguntas rapidamente.

Ejemplo: si te dan un arreglo fijo y $10^5$ consultas de minimo en rango, no
conviene recorrer el rango completo en cada consulta. Conviene construir una
tabla de respuestas parciales.

\subsection{Query o consulta}
Una query es una pregunta que se hace sobre la estructura ya construida. En
estos temas las queries tipicas son:
\begin{itemize}[leftmargin=*]
\item ``cual es el minimo de $A[l..r]$?''
\item ``cual es el LCA de $u$ y $v$?''
\item ``cuantas veces aparece el patron $P$ en el texto $S$?''
\item ``cual es el LCP de dos sufijos?''
\end{itemize}

\subsection{Operacion idempotente}
Una operacion es idempotente si repetir un elemento no cambia la respuesta:
\[
\min(x,x)=x,\qquad \max(x,x)=x,\qquad \gcd(x,x)=x.
\]
Esto es fundamental para sparse table. En una consulta de minimo podemos cubrir
un intervalo con dos bloques que se solapan, porque si un elemento queda contado
dos veces no altera el minimo. Para suma esto no funciona: $x+x\ne x$.

\subsection{Sparse table}
Una \textbf{sparse table} es una tabla de preprocesamiento para arreglos
estaticos. Guarda respuestas de rangos cuya longitud es potencia de dos:
\[
1,2,4,8,16,\ldots
\]
No guarda todos los rangos posibles, porque eso seria $\bigO(n^2)$. Guarda solo
rangos ``canonicos'':
\[
st[k][i] = \text{respuesta del rango que empieza en }i\text{ y mide }2^k.
\]
Para minimos:
\[
st[k][i]=\min(A[i],A[i+1],\ldots,A[i+2^k-1]).
\]

La razon por la que sirve es que cualquier rango $[l,r]$ contiene una potencia
de dos grande. Si $2^k\le r-l+1$, entonces dos bloques de longitud $2^k$ cubren
todo el intervalo: uno pegado a la izquierda y otro pegado a la derecha.

\subsection{Tabla de logs}
La sparse table suele necesitar:
\[
log[len]=\lfloor \log_2(len)\rfloor.
\]
Esto se puede precomputar con:
\[
log[1]=0,\qquad log[i]=log[\lfloor i/2\rfloor]+1.
\]
En papel no necesitas codigo, pero si debes saber que obtener $k$ para una
consulta cuesta $\bigO(1)$ si ya tienes esta tabla.

\subsection{Arbol enraizado, ancestro y profundidad}
Un arbol enraizado es un arbol donde escoges un nodo como raiz. Desde ahi se
define:
\begin{itemize}[leftmargin=*]
\item \textbf{padre}: el nodo anterior en el camino hacia la raiz;
\item \textbf{ancestro}: un nodo que aparece al subir desde $v$ hacia la raiz;
\item \textbf{profundidad}: numero de aristas desde la raiz hasta el nodo;
\item \textbf{subarbol}: un nodo y todos sus descendientes.
\end{itemize}

\subsection{LCA}
El \textbf{Lowest Common Ancestor} de $u$ y $v$ es el ancestro comun de ambos
que esta mas abajo, es decir, el de mayor profundidad. Si uno de los nodos es
ancestro del otro, entonces el LCA es ese mismo nodo.

Por que aparece tanto: muchas preguntas sobre caminos en arbol se parten en dos
tramos usando el LCA:
\[
u\to \LCA(u,v)\quad\text{y}\quad \LCA(u,v)\to v.
\]

\subsection{Binary lifting}
\textbf{Binary lifting} significa subir en potencias de dos. En lugar de guardar
solo el padre inmediato de cada nodo, guardas:
\[
up[v][0]=\text{padre de }v,
\]
\[
up[v][1]=\text{ancestro }2\text{ niveles arriba},
\]
\[
up[v][2]=\text{ancestro }4\text{ niveles arriba},
\]
y en general:
\[
up[v][j]=\text{ancestro }2^j\text{ niveles arriba de }v.
\]

La formula de construccion es:
\[
up[v][j]=up[\,up[v][j-1]\,][j-1].
\]
Se lee asi: para subir $2^j$, sube primero $2^{j-1}$ y desde ahi sube otros
$2^{j-1}$.

La idea funciona porque cualquier numero $k$ se puede escribir como suma de
potencias de dos. Por ejemplo, para subir $13$ niveles:
\[
13=8+4+1.
\]
Entonces saltas $8$, luego $4$, luego $1$.

\subsection{Euler tour}
Un \textbf{Euler tour} de un arbol enraizado es la secuencia de nodos visitados
por un DFS cuando entras y cuando vuelves de hijos. Para LCA no se guarda solo
la primera visita: se guarda la caminata completa. Tambien se guarda la
profundidad de cada visita.

Ejemplo conceptual: si al recorrer el arbol pasas por nodos
\[
E=[0,1,3,1,4,1,0,2,0],
\]
entonces guardas tambien:
\[
D=[0,1,2,1,2,1,0,1,0].
\]
El LCA de dos nodos se convierte en buscar la minima profundidad entre sus
primeras apariciones.

\subsection{Rank en strings}
En suffix array, \textbf{rank[i]} significa la posicion lexicografica del sufijo
que empieza en $i$. Si $rank[i]<rank[j]$, entonces:
\[
S[i..] < S[j..].
\]
El rank es el inverso del suffix array:
\[
\SA[rank[i]]=i.
\]

\subsection{Suffix array}
El \textbf{suffix array} es un arreglo que lista todos los sufijos de un string
en orden lexicografico. Si:
\[
S=\texttt{banana\$},
\]
sus sufijos son:
\[
\texttt{banana\$},\texttt{anana\$},\texttt{nana\$},\texttt{ana\$},
\texttt{na\$},\texttt{a\$},\texttt{\$}.
\]
Ordenados lexicograficamente quedan:
\[
\texttt{\$},\texttt{a\$},\texttt{ana\$},\texttt{anana\$},
\texttt{banana\$},\texttt{na\$},\texttt{nana\$}.
\]
El suffix array guarda sus posiciones iniciales:
\[
\SA=[6,5,3,1,0,4,2].
\]

\subsection{LCP}
\textbf{LCP} significa longest common prefix: prefijo comun mas largo. Por
ejemplo:
\[
\operatorname{lcp}(\texttt{anana\$},\texttt{ana\$})=3
\]
porque ambas empiezan con \texttt{ana}.

El arreglo LCP del suffix array guarda el LCP entre sufijos vecinos en orden
lexicografico. Esto es poderoso porque muchos problemas de substrings repetidas
se vuelven preguntas sobre vecinos o minimos en rangos del LCP.

\subsection{Sentinela}
El sentinela \texttt{\$} es un caracter artificial menor que todos los demas y
se pone al final del string. Sirve para que todos los sufijos sean comparables
sin casos ambiguos. Tambien asegura que ningun sufijo sea igual a otro.

\subsection{Separadores entre strings}
Cuando concatenas dos strings, por ejemplo para longest common substring, usas
separadores que no aparecen en ningun string:
\[
T=A\#B\$.
\]
Esto evita que una subcadena cruce artificialmente del final de $A$ al inicio de
$B$.

\subsection{Como decidir rapido que definicion necesitas}
\begin{center}
\begin{tabular}{p{0.34\linewidth}p{0.52\linewidth}}
\toprule
Palabra del enunciado & Definicion que debes activar \\
\midrule
rango estatico & sparse table, RMQ, operacion idempotente \\
ancestro, camino en arbol & LCA, profundidad, binary lifting \\
subir $k$ niveles & binary lifting, level ancestor \\
subarbol & Euler tour con tiempos de entrada/salida \\

substring repetida & LCP, vecinos en suffix array \\

\bottomrule
\end{tabular}
\end{center}

\section{RMQ en arrays estaticos}

\subsection{Definicion}
El problema \textbf{Range Minimum Query} recibe un arreglo fijo $A[0..n-1]$.
Una consulta $\RMQ(l,r)$ pide el indice o valor minimo en:
\[
A[l], A[l+1], \ldots, A[r].
\]
El slide de RMQ enfatiza tres metas:
\begin{itemize}[leftmargin=*]
\item construccion $\bigO(n\log n)$ y query $\bigO(1)$;
\item construccion $\bigO(n)$ y query $\bigO(\log n)$;
\item construccion $\bigO(n)$ y query $\bigO(1)$.
\end{itemize}

\subsection{Sparse table: la version que debes dominar}
La sparse table es una matriz donde las filas representan tamanos de bloque y
las columnas representan posiciones iniciales. La fila $0$ guarda bloques de
tamano $1$, la fila $1$ bloques de tamano $2$, la fila $2$ bloques de tamano
$4$, y asi sucesivamente.

La sparse table guarda respuestas para bloques de longitud potencia de dos:
\[
st[k][i]=\min(A[i],\ldots,A[i+2^k-1]).
\]
La transicion:
\[
st[k][i]=\min(st[k-1][i], st[k-1][i+2^{k-1}]).
\]

\subsubsection*{Consulta}
Para responder $[l,r]$:
\[
k=\lfloor \log_2(r-l+1)\rfloor.
\]
Usas dos bloques de longitud $2^k$:
\[
[l,l+2^k-1]\quad\text{y}\quad[r-2^k+1,r].
\]
La respuesta:
\[
\min(st[k][l], st[k][r-2^k+1]).
\]

Textualmente: tomo el bloque mas grande de potencia de dos que cabe dentro de
la consulta. Pongo una copia de ese bloque empezando en $l$ y otra terminando en
$r$. Entre ambos cubren todo $[l,r]$. Como estoy usando minimo, no importa si se
superponen.

\subsubsection*{Por que funciona}
Los dos bloques cubren todo el intervalo. Pueden solaparse, pero eso no importa
porque $\min$ es idempotente:
\[
\min(x,x)=x.
\]
Este argumento es el que debes escribir en examen.

\subsubsection*{Complejidad}
\[
\text{Preproceso } \bigO(n\log n), \quad
\text{query } \bigO(1), \quad
\text{espacio } \bigO(n\log n).
\]

\subsection{Cuando sparse table NO sirve}
Sparse table no soporta updates faciles. Si el problema dice:
\begin{itemize}[leftmargin=*]
\item cambia $A[i]$;
\item suma en rango;
\item asigna valores;
\end{itemize}
entonces piensa en segment tree, no en sparse table. Sparse table es para
datos estaticos y operaciones idempotentes como min/max/gcd.

\subsection{Como escribir sparse table en una solucion teorica}
Una respuesta fuerte en papel puede decir:
\begin{enumerate}[leftmargin=*]
\item Preproceso $st[k][i]$, el minimo del subarreglo de longitud $2^k$ que
empieza en $i$.
\item Caso base: $st[0][i]=A[i]$.
\item Transicion: parto el bloque de longitud $2^k$ en dos bloques de longitud
$2^{k-1}$.
\item Para consultar $[l,r]$, tomo $k=\lfloor\log_2(r-l+1)\rfloor$ y combino
los dos bloques extremos.
\item Correctitud: los bloques cubren el intervalo y el solapamiento no afecta
por idempotencia del minimo.
\item Complejidad: $\bigO(n\log n)$ construccion, $\bigO(1)$ query,
$\bigO(n\log n)$ memoria.
\end{enumerate}

\subsection{RMQ lineal: macro/micro bloques}
Los slides de static trees mencionan la meta optima $\bigO(n)$ espacio y
$\bigO(1)$ query. La idea:
\begin{enumerate}[leftmargin=*]
\item Partir el arreglo en bloques de tamano $b=\frac12\log n$.
\item Macroestructura: sparse table sobre minimos de bloques.
\item Microestructura: precomputar respuestas dentro de cada forma de bloque.
\end{enumerate}
En $\pm 1$ RMQ, cada bloque tiene pocas formas posibles: $2^{b-1}=O(\sqrt n)$.
Por eso se pueden precomputar tablas internas en $o(n)$ espacio.

\subsection{Hint de clasificacion}
Si ves:
\begin{itemize}[leftmargin=*]
\item ``static RMQ'';
\item ``range minimum, no updates'';
\item ``preprocess once, many queries'';
\end{itemize}
usa sparse table salvo que pidan explicitamente espacio lineal. Si piden
optimalidad teorica, menciona macro/micro.

\section{Cartesian trees: puente entre RMQ y LCA}

\subsection{Definicion}
El Cartesian tree de un arreglo $A$ es un arbol binario que cumple:
\begin{itemize}[leftmargin=*]
\item propiedad de heap: la raiz es el minimo del arreglo;
\item inorder traversal preserva el orden de indices;
\item el subarbol izquierdo se construye con elementos a la izquierda del minimo;
\item el subarbol derecho con elementos a la derecha.
\end{itemize}

\subsection{Identidad clave}
Para indices $i\le j$:
\[
\RMQ_A(i,j) = \LCA_T(i,j)
\]
donde $T$ es el Cartesian tree.

\subsection{Correctitud}
El minimo de $A[i..j]$ es el elemento de menor valor entre esos indices. En el
Cartesian tree, por la propiedad heap, ese elemento queda por encima de todos
los elementos del intervalo. Por la propiedad inorder, separa exactamente el
intervalo en izquierda y derecha. Por eso es el ancestro comun mas bajo de los
nodos $i$ y $j$.

\subsection{Construccion lineal}
El slide menciona el algoritmo con \textbf{right spine}:
\begin{enumerate}[leftmargin=*]
\item Escanear el arreglo de izquierda a derecha.
\item Mantener la espina derecha del Cartesian tree parcial.
\item Para insertar $A[i]$, subir por la espina hasta encontrar un nodo con valor menor.
\item El antiguo subarbol desplazado pasa a ser hijo izquierdo del nuevo nodo.
\end{enumerate}
Cada nodo entra y sale de la espina a lo mucho una vez, por eso el costo es
amortizado $\bigO(n)$.

\subsection{Hint de clasificacion}
Si un problema te pide demostrar equivalencia entre RMQ y LCA, o construir una
estructura optima para RMQ, menciona Cartesian tree. No es la primera herramienta
para implementar rapido, pero si es una herramienta teorica fuerte.

\section{LCA y su reduccion a RMQ}

\subsection{Definicion}
El \textbf{Lowest Common Ancestor} de dos nodos $u,v$ en un arbol enraizado es
el ancestro comun mas profundo de ambos.

Ejemplo textual: si el camino desde la raiz hasta $u$ es
\[
raiz\to a\to b\to u
\]
y el camino desde la raiz hasta $v$ es
\[
raiz\to a\to c\to v,
\]
entonces los ancestros comunes son $raiz$ y $a$. El mas profundo es $a$, por lo
tanto $\LCA(u,v)=a$.

\subsection{Por que LCA resuelve caminos}
En un arbol hay un unico camino simple entre dos nodos. Si quieres responder
algo sobre el camino entre $u$ y $v$, el LCA es el punto donde ese camino se
parte:
\[
u \to \LCA(u,v) \to v.
\]
Por eso LCA aparece en distancia, maximo peso en camino, minimo peso en camino,
numero de aristas, ancestros y problemas tipo ``Imperial Roads''.

\subsection{Solucion simple: binary lifting}
Preprocesas:
\[
up[v][j] = \text{ancestro }2^j\text{ de }v.
\]
Esto es una tabla de saltos. La columna $0$ guarda saltos de una arista; la
columna $1$ guarda saltos de dos aristas; la columna $2$ guarda saltos de
cuatro aristas; y asi sucesivamente.

La construccion se basa en:
\[
up[v][0]=parent[v],
\]
\[
up[v][j]=up[\,up[v][j-1]\,][j-1].
\]
Se lee asi: para subir $2^j$ niveles desde $v$, primero subes $2^{j-1}$ niveles
y desde ese nodo vuelves a subir $2^{j-1}$ niveles.

Para responder:
\begin{enumerate}[leftmargin=*]
\item Si un nodo esta mas profundo que el otro, lo subes hasta que ambos tengan
la misma profundidad.
\item Si despues de igualarlos son el mismo nodo, ese nodo era ancestro del otro
y ya es la respuesta.
\item Si son distintos, subes ambos al mismo tiempo desde potencias grandes a
pequenas.
\item Cuando $up[u][j]\ne up[v][j]$, puedes subir ambos $2^j$ niveles sin pasar
por encima del LCA.
\item Al final, $u$ y $v$ quedan justo debajo del LCA; entonces $parent[u]$ es
la respuesta.
\end{enumerate}
Complejidad:
\[
\bigO(n\log n)\text{ preproceso},\quad \bigO(\log n)\text{ query}.
\]

\subsection{Correctitud de binary lifting}
La primera fase no cambia el LCA: si un nodo esta mas abajo, los vertices que
estan estrictamente debajo de la profundidad del otro nodo no pueden ser
ancestros comunes. Por eso se pueden eliminar subiendo.

La segunda fase mantiene el invariante de que $u$ y $v$ siguen debajo del LCA,
pero en ramas distintas. Se prueban saltos grandes primero para acercarse lo mas
posible al LCA. Si un salto llevara a ambos al mismo ancestro, ese salto alcanza
o pasa el LCA, asi que no se toma. Cuando ya no hay salto valido, los padres de
ambos coinciden y ese padre es el LCA.

\subsection{Binary lifting para subir k niveles}
La misma tabla resuelve \textbf{Level Ancestor}. Para subir $k$ niveles desde
$v$, escribes $k$ como suma de potencias de dos. Por ejemplo:
\[
13=8+4+1.
\]
Entonces haces:
\[
v\leftarrow up[v][3],\quad v\leftarrow up[v][2],\quad v\leftarrow up[v][0].
\]
Cada bit prendido de $k$ representa un salto. Esto cuesta $\bigO(\log n)$.

\subsection{Reduccion a RMQ con Euler tour}
Haces un Euler tour del arbol. Guardas:
\begin{itemize}[leftmargin=*]
\item $E$: secuencia de nodos visitados;
\item $D$: profundidad de cada visita;
\item $first[v]$: primera posicion de $v$ en $E$.
\end{itemize}
Entonces:
\[
\LCA(u,v)=E[\arg\min D[l..r]]
\]
donde:
\[
l=\min(first[u],first[v]),\quad r=\max(first[u],first[v]).
\]

\subsection{Propiedad $\pm 1$}
En el Euler tour, cada paso sube o baja una arista. Por eso:
\[
|D[i]-D[i-1]|=1.
\]
El problema LCA se reduce a $\pm 1$ RMQ, que puede resolverse en $\bigO(n)$
espacio y $\bigO(1)$ query con macro/micro.

Que significa ``$\pm1$ RMQ'': es un RMQ sobre un arreglo donde dos valores
consecutivos difieren exactamente en uno. En el Euler tour, la profundidad sube
o baja una arista en cada paso, por eso aparece esta propiedad. Esa restriccion
permite usar bloques pequenos y precomputar respuestas segun la forma del bloque.

\subsection{Correctitud}
Entre la primera aparicion de $u$ y la primera aparicion de $v$ en el Euler tour,
el recorrido pasa por todos los nodos necesarios para ir de uno al otro en el
DFS. El nodo de profundidad minima en ese segmento es exactamente el ancestro
comun mas bajo. Cualquier ancestro comun esta por encima; el mas bajo es el de
menor profundidad que aparece entre ambos.

\subsection{Hint de clasificacion}
Si ves:
\begin{itemize}[leftmargin=*]
\item distancia en arbol;
\item camino entre dos nodos;
\item ancestro comun;
\item max/min edge on path con arbol fijo;
\end{itemize}
probablemente necesitas LCA. Para distancias:
\[
\operatorname{dist}(u,v)=depth[u]+depth[v]-2depth[\LCA(u,v)].
\]

\subsection{Como decidir entre las dos soluciones de LCA}
\begin{itemize}[leftmargin=*]
\item Si quieres una solucion sencilla de explicar: binary lifting.
\item Si el problema conecta explicitamente LCA con RMQ: Euler tour + RMQ.
\item Si piden query $\bigO(1)$ con espacio lineal: Euler tour + $\pm1$ RMQ
macro/micro.
\item Si hay pesos en aristas y consultas de maximo/minimo en camino: binary
lifting puede guardar tambien el maximo/minimo de cada salto.
\end{itemize}

\section{Strings: notacion basica}

\subsection{Conceptos}
Los slides definen:
\begin{itemize}[leftmargin=*]
\item Alfabeto $\Sigma$.
\item Cadena $S$ de longitud $|S|$.
\item Subcadena $S[i,j]$.
\item Prefijo $\Pref(S,i)=S[0,i]$.
\item Sufijo $\Suf(S,i)=S[i,|S|-1]$.
\item LCP: prefijo comun mas largo entre dos cadenas.
\item Sentinela \texttt{\$}: caracter menor que todos que aparece al final.
\end{itemize}

\subsection{Subcadena, subsecuencia, prefijo y sufijo}
Estos terminos se parecen, pero no significan lo mismo:
\begin{itemize}[leftmargin=*]
\item \textbf{Subcadena} o substring: bloque contiguo. En \texttt{banana},
\texttt{ana} es subcadena.
\item \textbf{Subsecuencia}: caracteres en orden, pero no necesariamente
contiguos. En \texttt{banana}, \texttt{bnn} es subsecuencia, no subcadena.
\item \textbf{Prefijo}: subcadena que empieza en la posicion $0$.
\item \textbf{Sufijo}: subcadena que termina en la ultima posicion.
\end{itemize}
Suffix array trabaja con sufijos, pero sirve para subcadenas porque toda
subcadena $S[l..r]$ es prefijo del sufijo $S[l..]$.

\subsection{Por que importa el sentinela}
El sentinela evita casos molestos donde un sufijo es prefijo de otro. Si
\texttt{\$} es menor que todo, entonces las comparaciones lexicograficas son
totales y limpias.

\subsection{Orden lexicografico}
Dos cadenas se comparan por el primer caracter donde difieren. Si una termina
antes y es prefijo propio de la otra, la mas corta es menor. Esta definicion es
la base de suffix array.

\subsection{Idea clave para problemas}
Si una subcadena $P$ aparece en $S$ empezando en la posicion $i$, entonces el
sufijo $S[i..]$ empieza con $P$. Por eso buscar una subcadena se transforma en
buscar sufijos que tienen cierto prefijo.

\subsection{Hint de clasificacion}
Si el problema habla de:
\begin{itemize}[leftmargin=*]
\item patron;
\item subcadena;
\item sufijo;
\item prefijo comun;
\item orden lexicografico;
\item substring repetida;
\end{itemize}
debes pensar inmediatamente en suffix array, LCP, KMP/Z, o Aho-Corasick.

\section{Suffix array}

\subsection{Definicion}
El suffix array de $S$ es una permutacion $A$ de posiciones:
\[
\Suf(S,A_i) < \Suf(S,A_{i+1})
\]
para todo $i$. Guarda exactamente $n$ enteros, por eso es mas compacto que un
suffix tree.

Mas textualmente: escribes todos los sufijos de $S$, los ordenas como palabras
de diccionario y guardas solo el indice donde empieza cada sufijo. No guardas
las cadenas completas.

Ejemplo:
\[
S=\texttt{banana\$}.
\]
Los sufijos son:
\[
\begin{array}{c|l}
0 & \texttt{banana\$}\\
1 & \texttt{anana\$}\\
2 & \texttt{nana\$}\\
3 & \texttt{ana\$}\\
4 & \texttt{na\$}\\
5 & \texttt{a\$}\\
6 & \texttt{\$}
\end{array}
\]
Ordenados:
\[
\begin{array}{c|l}
6 & \texttt{\$}\\
5 & \texttt{a\$}\\
3 & \texttt{ana\$}\\
1 & \texttt{anana\$}\\
0 & \texttt{banana\$}\\
4 & \texttt{na\$}\\
2 & \texttt{nana\$}
\end{array}
\]
Entonces:
\[
\SA=[6,5,3,1,0,4,2].
\]

\subsection{Por que sirve}
Ordenar sufijos convierte busqueda de patrones en busqueda binaria:
\begin{itemize}[leftmargin=*]
\item Un patron $P$ aparece donde algun sufijo empieza con $P$.
\item Todos esos sufijos estan consecutivos en el suffix array.
\item Por tanto, las ocurrencias son un intervalo en $\SA$.
\end{itemize}

La frase ``intervalo en el suffix array'' es clave. Si todos los sufijos que
empiezan con \texttt{ana} estan ordenados lexicograficamente, no pueden quedar
separados por un sufijo que no empieza con \texttt{ana}. Todos comparten el
mismo prefijo, asi que forman un bloque contiguo.

\subsection{Rank inverso}
Ademas de $\SA$, normalmente se guarda:
\[
rank[i]=\text{posicion del sufijo }S[i..]\text{ dentro de }\SA.
\]
Si $\SA=[6,5,3,1,0,4,2]$, entonces $rank[3]=2$ porque el sufijo que empieza en
$3$ aparece en la posicion $2$ del suffix array. Esto permite comparar sufijos
en $\bigO(1)$: menor rank significa menor sufijo.

\subsection{Aplicaciones de examen}
\begin{itemize}[leftmargin=*]
\item Buscar si $P$ aparece en $S$.
\item Contar ocurrencias de $P$.
\item Listar posiciones donde aparece $P$.
\item Contar subcadenas distintas.
\item Longest common substring.
\item Comparar/ordenar substrings.
\item Encontrar repeticiones usando LCP.
\end{itemize}

\subsection{Hint de clasificacion}
Si el texto es fijo y hay muchas consultas de patrones, usa suffix array. Si hay
muchos patrones y quieres procesarlos todos simultaneamente, Aho-Corasick puede
ser mejor. Si necesitas ordenar substrings, suffix array + LCP + RMQ.

\section{Manber--Myers: construccion $\bigO(n\log n)$}

\subsection{Idea central}
Manber--Myers ordena sufijos por prefijos de longitud creciente:
\[
1,2,4,8,\ldots
\]
Mantiene $rank[i]$: clase lexicografica del prefijo de longitud actual que
empieza en $i$.

Al inicio solo sabes ordenar por el primer caracter. Despues usas esa
informacion para ordenar por dos caracteres, luego por cuatro, luego por ocho.
Cuando la longitud considerada supera $n$, ordenar esos prefijos equivale a
ordenar los sufijos completos.

\subsection{Que es una clase o rank temporal}
Dos posiciones tienen el mismo rank temporal si sus prefijos de la longitud
actual son iguales. Por ejemplo, en la primera ronda, todas las posiciones con
caracter \texttt{a} tienen el mismo rank, todas las posiciones con
\texttt{b} otro rank, etc. En rondas posteriores el rank representa bloques mas
largos.

\subsection{Paso inductivo}
Si ya conoces ranks para longitud $2^k$, entonces el prefijo de longitud
$2^{k+1}$ de $i$ queda descrito por:
\[
(rank[i], rank[i+2^k]).
\]
Ordenar estos pares ordena por longitud doble.

\subsection{Pseudocodigo teorico}
\begin{enumerate}[leftmargin=*]
\item Agrega sentinela.
\item Ordena posiciones por caracter y asigna ranks iniciales.
\item Para $len=1,2,4,\ldots$:
\begin{enumerate}
\item Ordena posiciones por $(rank[i],rank[i+len])$.
\item Reasigna nuevos ranks.
\end{enumerate}
\item Cuando $len\ge n$, el orden es el suffix array.
\end{enumerate}

\subsection{Optimizacion del slide}
El slide explica que se puede evitar ordenar por el segundo componente desde
cero. Como las posiciones ya estan ordenadas por la segunda mitad, se desplazan
indices y se usa counting sort estable por el primer rank. Esto mantiene
$\bigO(n)$ por ronda.

\subsection{Correctitud}
Por induccion. En la ronda $k$, $rank[i]$ representa correctamente el orden del
prefijo de longitud $2^k$. Dos prefijos de longitud $2^{k+1}$ se comparan por
su primera mitad; si empatan, por su segunda mitad. Eso es exactamente comparar
el par de ranks.

\subsection{Complejidad}
$\bigO(\log n)$ rondas. Cada una $\bigO(n)$ con counting/radix sort. Total:
\[
\bigO(n\log n)
\]
tiempo y $\bigO(n)$ o $\bigO(n\log n)$ espacio segun si guardas todos los niveles.

\section{LCP array y RMQ sobre LCP}

\subsection{Definicion}
Para suffix array $\SA$:
\[
\LCP[i]=\operatorname{lcp}(S[\SA[i-1]..],S[\SA[i]..]).
\]
Mide cuanto comparten sufijos vecinos.

Ejemplo con \texttt{banana\$}: los sufijos vecinos \texttt{ana\$} y
\texttt{anana\$} tienen LCP $3$, porque comparten \texttt{ana}. Los sufijos
\texttt{na\$} y \texttt{nana\$} tienen LCP $2$, porque comparten \texttt{na}.

\subsection{Por que solo vecinos}
El LCP array guarda LCP entre vecinos porque el orden lexicografico junta sufijos
parecidos. Si quieres detectar repeticiones largas, normalmente aparecen como
LCP grande entre sufijos cercanos. Si quieres el LCP de sufijos no vecinos, lo
obtienes con RMQ sobre los LCP intermedios.

\subsection{LCP arbitrario entre dos sufijos}
Si los sufijos que empiezan en $i$ y $j$ estan en posiciones $a$ y $b$ del
suffix array, con $a<b$, entonces:
\[
\operatorname{lcp}(i,j)=\min(\LCP[a+1],\ldots,\LCP[b]).
\]
Esto es una RMQ sobre el arreglo LCP.

Textualmente: para saber cuanto comparten dos sufijos lejanos en el suffix
array, miras el tramo entre ellos. El prefijo comun de todos los sufijos de ese
tramo esta limitado por el menor LCP entre vecinos. Por eso aparece un minimo.

\subsection{Correctitud}
Entre dos sufijos en orden lexicografico, todos los sufijos intermedios comparten
al menos el prefijo comun minimo del intervalo. El primer punto donde alguno
difiere queda capturado por el minimo LCP entre vecinos. Por eso el LCP de dos
sufijos cualesquiera es el minimo LCP en el rango entre sus posiciones.

\subsection{Usos}
\begin{itemize}[leftmargin=*]
\item Comparar substrings en $\bigO(1)$.
\item Encontrar repeticiones.
\item Longest common substring.
\item Ordenar substrings.
\item Resolver problemas donde muchas queries piden igualdad de substrings.
\end{itemize}

\subsection{Plantilla de examen para SA + LCP}
Cuando uses suffix array y LCP, escribe:
\begin{enumerate}[leftmargin=*]
\item Construyo $\SA$ para ordenar todos los sufijos.
\item Construyo $rank$ para saber la posicion de cada sufijo en $\SA$.
\item Construyo $\LCP$ entre sufijos vecinos.
\item Si necesito LCP arbitrario, preproceso RMQ sobre $\LCP$.
\item La correctitud viene de que el orden lexicografico agrupa sufijos con
prefijos comunes.
\item La complejidad depende de la construccion de $\SA$: $\bigO(n\log n)$ con
Manber--Myers o $\bigO(n)$ con DC3/SA-IS.
\end{enumerate}

\subsection{Hint de clasificacion}
Si ves comparaciones repetidas de substrings, no compares caracter por caracter.
Preprocesa suffix array, rank, LCP y RMQ.

\section{Pattern matching}

\subsection{Dos escenarios de los slides}
Los slides separan:
\begin{itemize}[leftmargin=*]
\item Patron conocido, texto por conocer: KMP, Z, Rabin-Karp, Aho-Corasick.
\item Texto conocido, patron por conocer: suffix array, suffix tree, suffix automaton.
\end{itemize}

\subsection{Texto fijo + patrones}
Construye suffix array de $S$. Para patron $P$, busca el intervalo de sufijos
que empiezan con $P$. Dos busquedas binarias bastan.

\subsection{Muchos patrones}
Si todos los patrones se conocen antes y quieres recorrer el texto una vez,
Aho-Corasick es natural. Si el examen esta en el bloque de suffix array, explica
la solucion con suffix array.

\subsection{Correctitud del intervalo}
Todos los sufijos con prefijo $P$ forman un bloque lexicografico. Si un sufijo
sin prefijo $P$ estuviera dentro, diferiria en algun caracter y romperia el orden
del bloque. Por eso el bloque encontrado por binary search contiene exactamente
las ocurrencias.

\subsection{Complejidad}
Simple:
\[
\bigO(n\log n)\text{ construir},\quad \bigO(|P|\log n)\text{ por patron}.
\]
Con LCP optimizado se puede mejorar, pero en papel la idea del intervalo es lo
mas importante.

\section{Contar subcadenas distintas}

\subsection{Formula}
Un string de longitud $n$ tiene:
\[
\frac{n(n+1)}2
\]
subcadenas contando repeticiones. Para contar distintas:
\[
\#distinct = \frac{n(n+1)}2 - \sum_i \LCP[i].
\]

\subsection{Por que funciona}
Cada sufijo aporta todos sus prefijos como subcadenas. Si procesas sufijos en
orden de suffix array, el sufijo actual comparte $\LCP[i]$ prefijos con el
anterior. Esos prefijos ya fueron contados. Por tanto, el sufijo aporta:
\[
(n-\SA[i])-\LCP[i]
\]
nuevas subcadenas.

\subsection{Hint de clasificacion}
Si el problema pide:
\begin{itemize}[leftmargin=*]
\item numero de substrings distintas;
\item cuantas subcadenas diferentes tiene $S$;
\item contar repetidas vs unicas;
\end{itemize}
usa suffix array + LCP y esta formula.

\subsection{Complejidad}
Construir $\SA+\LCP$ domina. Luego sumar LCP es $\bigO(n)$.

\section{Longest common substring}

\subsection{Problema}
Dadas cadenas $A$ y $B$, encontrar la subcadena mas larga que aparece en ambas.

\subsection{Algoritmo}
Concatena:
\[
T=A\#B\$
\]
con separadores que no aparecen en las cadenas. Construye $\SA$ y $\LCP$ de $T$.
Recorre pares vecinos en $\SA$. Si un sufijo viene de $A$ y el otro de $B$,
actualiza respuesta con su LCP.

\subsection{Correctitud}
Si una subcadena $X$ aparece en ambas cadenas, entonces hay un sufijo de $A$ y
uno de $B$ que tienen prefijo $X$. Todos los sufijos con prefijo $X$ estan juntos
en $\SA$. Dentro de ese bloque debe haber algun par vecino de origen distinto.
Su LCP es al menos $|X|$.

\subsection{Complejidad}
$\bigO(n\log n)$ con Manber--Myers o $\bigO(n)$ con DC3/SA-IS para construir.
Recorrido final $\bigO(n)$.

\section{Ordenar y comparar subcadenas}

\subsection{Problema}
Muchas veces te dan consultas con substrings $S[l,r]$ y debes compararlas,
ordenarlas o decidir igualdad.

\subsection{Estructura}
Preprocesa:
\begin{itemize}[leftmargin=*]
\item $\SA$;
\item rank inverso de cada sufijo;
\item $\LCP$;
\item RMQ sobre $\LCP$.
\end{itemize}

\subsection{Comparacion}
Para comparar $S[l_1,r_1]$ y $S[l_2,r_2]$:
\begin{enumerate}[leftmargin=*]
\item Calcula $x=\operatorname{lcp}(l_1,l_2)$ con RMQ sobre LCP.
\item Si $x$ cubre la longitud menor, gana la substring mas corta.
\item Si no, compara $S[l_1+x]$ y $S[l_2+x]$.
\end{enumerate}

\subsection{Complejidad}
Preprocesamiento $\bigO(n\log n)$ con sparse table sobre LCP. Cada comparacion
$\bigO(1)$. Ordenar $q$ substrings cuesta $\bigO(q\log q)$ comparaciones.

\section{DC3}

\subsection{Idea}
DC3 construye suffix array en tiempo lineal. Divide posiciones por residuo
modulo 3:
\[
S_0=\{i:i\equiv0\},\quad S_1=\{i:i\equiv1\},\quad S_2=\{i:i\equiv2\}.
\]
Primero ordena recursivamente sufijos de $S_1\cup S_2$. Luego deduce $S_0$ y
hace merge.

\subsection{Por que sirve}
El teorema del slide dice: si sabes el orden de sufijos en $S_1$ y $S_2$,
puedes deducir el orden de todos. Para comparar un sufijo de $S_0$ con otro,
usas caracteres iniciales y ranks ya conocidos.

\subsection{Pasos}
\begin{enumerate}[leftmargin=*]
\item Formar ternas de caracteres para posiciones en $S_1$ y $S_2$.
\item Ordenar ternas con radix sort y asignar ranks.
\item Si hay empates, resolver recursivamente la cadena de ranks.
\item Ordenar $S_0$ usando $(S[i],rank[i+1])$.
\item Mezclar linealmente $S_0$ con $S_1\cup S_2$.
\end{enumerate}

\subsection{Complejidad}
La recurrencia:
\[
T(n)=T(2n/3)+\bigO(n)
\]
da:
\[
T(n)=\bigO(n).
\]

\subsection{Hint de examen}
No intentes memorizar implementacion. Aprende:
\begin{itemize}[leftmargin=*]
\item particion modulo 3;
\item ordenar $2n/3$ posiciones recursivamente;
\item usar ranks para comparar en $\bigO(1)$;
\item recurrencia lineal.
\end{itemize}

\section{SA-IS y clasificacion S/L}

\subsection{Idea}
SA-IS clasifica sufijos en tipo S y tipo L:
\begin{itemize}[leftmargin=*]
\item tipo S si el sufijo actual es menor que el siguiente;
\item tipo L si es mayor que el siguiente;
\item LMS si hay transicion de L a S.
\end{itemize}

\subsection{Teorema importante}
Si sabes el orden relativo de los sufijos LMS, puedes inducir el orden de todos
los sufijos. Por eso SA-IS:
\begin{enumerate}[leftmargin=*]
\item clasifica S/L;
\item ordena LMS;
\item induce L de izquierda a derecha;
\item induce S de derecha a izquierda;
\item si hace falta, recurre sobre nombres de LMS substrings.
\end{enumerate}

\subsection{Por que es lineal}
Cada fase de induccion escanea buckets una cantidad constante de veces. El
problema recursivo tiene tamano a lo mucho el numero de LMS, que es a lo mucho
$n/2$. Por tanto el total es lineal.

\subsection{Hint de examen}
SA-IS es mas conceptual que implementable a papel. Si preguntan, escribe:
clasificacion S/L, LMS, induced sorting, recursion sobre LMS names, y
complejidad $\bigO(n)$.

\section{Bordes, KMP y periodicidad}

\subsection{Bordes}
Un borde de $S$ es una cadena que es prefijo y sufijo. La funcion prefijo de KMP:
\[
\pi[i]=\text{longitud del mayor borde de }S[0..i].
\]
Los bordes de todo $S$ se obtienen siguiendo:
\[
k=\pi[n-1],\quad k=\pi[k-1],\quad \ldots
\]

\subsection{Periodicidad}
$S$ tiene periodo $p$ si:
\[
S[i]=S[i+p]
\]
cuando ambas posiciones existen. Para substrings, puedes verificar igualdad de
zonas con LCP:
\[
\operatorname{lcp}(l,l+p)\ge len-p.
\]

\subsection{Hint de clasificacion}
Si dice:
\begin{itemize}[leftmargin=*]
\item borde;
\item prefijo que tambien es sufijo;
\item periodo;
\item repeticion exacta;
\end{itemize}
piensa primero en KMP/Z. Si hay muchas consultas sobre substrings arbitrarias,
combina con suffix array + LCP.
